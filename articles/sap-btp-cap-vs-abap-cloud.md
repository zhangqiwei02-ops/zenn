# CAP と ABAP Cloud の違いを整理する

SAP BTP の拡張開発を検討するとき、よく話題になるのが **CAP** と **ABAP Cloud** のどちらを選ぶべきかという点です。

この議論がややこしくなるのは、両者を「どちらが先進的か」「どちらが正しいか」という構図で見てしまうからです。実際には、比較すべきなのは技術の優劣ではなく、**どの責務をどこに置くべきか** という設計上の境界です。

この記事では、CAP と ABAP Cloud を SAP BTP 時代の拡張開発という文脈で比較し、どこで線を引くべきかを整理します。

## 先に結論

結論から言うと、基本の考え方は次の通りです。

- **S/4HANA の業務オブジェクトやロジックに近い拡張** → **ABAP Cloud**
- **独立性の高いサービス、周辺アプリ、外部連携を含む拡張** → **CAP**

つまり、

- コア業務に近い責務を保守的に扱うなら ABAP Cloud
- 境界を切り出しやすく、クラウドネイティブに進化させたいなら CAP

です。

ここで重要なのは、どちらが優れているかではなく、**変更責任をどこに置くと無理がないか** で判断することです。

## なぜ混同されやすいのか

CAP も ABAP Cloud も、SAP の拡張開発を支える選択肢である点は共通しています。そのため、表面的には似て見えます。

ですが、実際には設計の重心が違います。

- CAP は、BTP 上で独立したサービスや API を構成しやすい
- ABAP Cloud は、SAP 業務モデルに近い場所での拡張と整合性に強い

この違いを曖昧にしたまま「好み」で選ぶと、後から責任境界が崩れます。

## CAP が向くケース

CAP は、BTP 上で比較的独立したドメインサービスやアプリケーションを構築したい場合に強いです。

### 向いているケース

- 外部 SaaS や他システムとの連携が多い
- API 中心で設計したい
- イベントやワークフローも含めて構成したい
- クラウドネイティブな CI/CD やテストを重視したい
- Node.js / Java 系のチームが中心

### 強み

- サービス境界を切りやすい
- API / Event Mesh / Integration Suite との相性がよい
- 周辺機能を柔軟に拡張しやすい
- SAP 外のアーキテクチャとも会話がしやすい

### 注意点

- S/4HANA の業務文脈との距離が出やすい
- コア業務の責務を外に持ち出しすぎると整合性管理が重くなる

## ABAP Cloud が向くケース

ABAP Cloud は、S/4HANA の業務モデルに近い場所で、安全に拡張を行いたい場合に向いています。

### 向いているケース

- 標準オブジェクトや業務ロジックとの整合性が重要
- Clean Core を意識した拡張を行いたい
- 既存 ABAP チームの知見を活かしたい
- アップグレード耐性を重視したい
- SAP 標準の拡張作法に沿いたい

### 強み

- 業務ロジックとの距離が近い
- SAP の文脈で責務を保ちやすい
- ABAP 資産や人材を活かしやすい
- 標準拡張モデルとの相性がよい

### 注意点

- 外部連携や周辺アプリの主戦場としては CAP より自由度が低く感じることがある
- 何でも内側で抱えると、境界整理を怠りやすい

## 比較表で整理する

| 観点 | CAP | ABAP Cloud |
|---|---|---|
| 主な役割 | 独立サービス / 周辺アプリ | コア業務に近い拡張 |
| S/4HANA との距離 | やや遠い | 近い |
| 外部連携 | 得意 | 可能だが主戦場ではない |
| チーム相性 | Node.js / Java / クラウドネイティブ | ABAP / SAP 業務知識 |
| 変更責任 | サービス単位で切りやすい | SAP 業務責務に沿って持ちやすい |

この表から分かるのは、両者が競合というより、扱う責務が違うということです。

## 何を軸に線を引くべきか

実務で特に重要なのは、次の軸です。

1. **S/4HANA との結合度**
2. **業務責務をどこに置くか**
3. **チームのスキルと運用モデル**
4. **テストとリリースをどう回すか**

例えば、受注や会計のように S/4HANA の業務文脈から切り離しづらいものは ABAP Cloud が自然です。一方で、外部システム連携や周辺 UX を独立して進化させたいなら CAP が扱いやすくなります。

## 典型シナリオで考える

### シナリオ 1: S/4HANA の受注処理に密接な拡張

この場合は、標準オブジェクトとの整合性とアップグレード耐性が重要です。

- → ABAP Cloud が有力

### シナリオ 2: BTP 上の申請アプリと外部 API 連携

この場合は、周辺サービスとして独立性を持たせた方が設計しやすいです。

- → CAP が有力

### シナリオ 3: コア業務と周辺アプリが混在する

この場合は、コア寄りは ABAP Cloud、周辺は CAP と役割分担する方が自然です。

実務では、この組み合わせがむしろ普通です。

## まとめ

CAP と ABAP Cloud は、どちらが正義かで選ぶものではありません。

- ABAP Cloud は、S/4HANA に近い責務を安全に拡張するための選択肢
- CAP は、独立性の高いサービスや周辺アプリを柔軟に作るための選択肢

判断するときは、技術の好みではなく、**境界・責任・保守のしやすさ** を見るべきです。

SAP BTP の拡張開発で本当に重要なのは、どの技術を信じるかではなく、どこに変更責任を置くと長期運用で壊れにくいかを見極めることです。
