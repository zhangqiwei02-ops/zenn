# 会社の NAS がパンパン？コード変更なしで S3 に繋ぐ方法を整理する

オンプレミスの NAS 容量不足に悩んだとき、よくあるのは「全部クラウドへ移行しないといけないのでは」と考えてしまうことです。

しかし実務では、既存の共有フォルダ運用やアプリケーションの前提を壊さずに、段階的にクラウドへ寄せたい場面が多くあります。そうしたときに候補になるのが **AWS Storage Gateway の File Gateway** です。

この記事では、File Gateway がなぜ「コード変更なしで NAS をクラウド拡張できる手段」になるのかを整理します。

## 結論：既存の共有ファイル運用を壊さずに S3 へ繋げたいなら File Gateway

先に結論を書くと、既存の SMB / NFS ベースの運用を変えずに Amazon S3 を活用したいなら、File Gateway がかなり有力です。

理由は次の通りです。

- クライアントからはファイル共有として見える
- 裏側では Amazon S3 と連携できる
- よく使うデータはローカルキャッシュで扱える
- 既存アプリケーションの大規模改修を避けやすい

つまり、**現場の使い方は大きく変えず、保存基盤だけをクラウドへ寄せる**という構成が取りやすくなります。

## なぜ S3 直利用ではなく Gateway なのか

S3 は優秀なオブジェクトストレージですが、従来の NAS のような使い方とは前提が異なります。

たとえば、社内システムや既存ツールが次を前提にしている場合、S3 へ直接切り替えるのは簡単ではありません。

- SMB 共有を前提にしている
- NFS マウントを前提にしている
- 共有フォルダのパス参照を前提にしている
- ファイルシステムとして扱う業務手順がある

このギャップを埋めるのが File Gateway です。

## File Gateway の役割

File Gateway は、オンプレミス側からは SMB / NFS の共有として見えつつ、保存先として Amazon S3 を利用できるサービスです。

このため、既存の NAS 利用に近い形でクラウド活用を進められます。

### 向いているケース
- 既存 NAS の容量不足を解消したい
- アプリケーション改修を最小限にしたい
- ハイブリッドクラウド構成を取りたい
- 保存先を Amazon S3 に寄せたい

ここが実務上の大きなメリットです。

## SMB / NFS をそのまま使える意味

File Gateway の価値は、技術的に S3 と繋がることだけではありません。現場への影響を抑えやすいことが重要です。

たとえば、

- Windows クライアントが SMB で使う
- Linux サーバーが NFS でマウントする
- 部門システムが共有フォルダを参照する

こうした既存利用を大きく壊さずに済むと、移行プロジェクト全体の摩擦が下がります。

つまり、File Gateway は単なるストレージ接続機能ではなく、**現場運用を壊さずにクラウドへ橋を架けるためのサービス**だと理解すると分かりやすいです。

## キャッシュと S3 の組み合わせ

File Gateway では、よく使うデータをローカルにキャッシュしながら、保存先として Amazon S3 を使えます。

このため、次のようなバランスが取りやすくなります。

- 頻繁に使うファイルはローカル寄りで扱う
- 古いファイルや大量データは S3 に逃がす
- オンプレミス機器の容��増設だけに依存しない

もちろん、すべてのワークロードに万能というわけではありません。極端に低レイテンシが必要な用途や、特殊な I/O 前提がある場合は事前検証が必要です。

## 向いているケース / 向いていないケース

### 向いているケース
- NAS 容量不足を段階的に解消したい
- 既存の SMB / NFS 運用を維持したい
- Amazon S3 を保存先にしたい
- コード変更を避けたい

### 向いていないケース
- 最初から完全クラウドネイティブに寄せる前提
- ブロックストレージ要件が強い
- ローカルディスク並みの低遅延を絶対条件とする

こうした切り分けを持っておくと、設計判断がかなり安定します。

## 試験での見分け方

AWS 認定試験では、次の条件が並んだら File Gateway を疑うべきです。

- オンプレミスのファイル共有を維持したい
- Amazon S3 を保存先にしたい
- アプリ改修を避けたい
- SMB / NFS を継続利用したい

逆に、オブジェクト API をそのまま使えるなら S3 直利用の方が自然です。ここを分けて覚えると、かなり迷いにくくなります。

## まとめ

オンプレミス NAS の容量不足に対して、毎回フル移行だけが正解とは限りません。

**AWS Storage Gateway の File Gateway** を使えば、

- 既存の SMB / NFS 運用を保ちやすい
- アプリケーション改修を避けやすい
- Amazon S3 の容量を活用できる

という形で、かなり現実的なハイブリッド拡張ができます。

「今の使い方を大きく壊さずに、クラウドへ寄せたい」。その条件なら、まず File Gateway を候補に入れる価値があります。
