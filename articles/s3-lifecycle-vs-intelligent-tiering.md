# S3 Lifecycle と Intelligent-Tiering の違いを整理する

S3 のコスト最適化を考えるとき、**S3 Lifecycle** と **S3 Intelligent-Tiering** を同じもののように扱ってしまうケースはかなり多いです。

ですが、この 2 つは「安くする仕組み」という点が共通しているだけで、何を基準に最適化するのかが違います。ここを曖昧にすると、試験では選択肢を外し、実務では取り出し遅延や想定外コストで困ります。

この記事では、Lifecycle と Intelligent-Tiering を **時間ベースのルール** と **アクセスベースの自動最適化** という軸で整理します。

## 先に結論

結論から言うと、役割は次のように分かれます。

- **S3 Lifecycle**: 経過日数などの条件でストレージクラス移行や削除を行う
- **S3 Intelligent-Tiering**: アクセスパターンを見ながら自動で適切な階層へ最適化する

つまり、

- 古くなる時期が読めるデータには Lifecycle
- 読まれ方が読めないデータには Intelligent-Tiering

が向いています。

ここで重要なのは、**Lifecycle は一度下げたオブジェクトをアクセス増加に応じて自動で Standard に戻す仕組みではない** という点です。

## なぜ混同されやすいのか

両方とも S3 の保存コストを下げるために使うため、表面上は似て見えます。

ただし、内部の発想はかなり違います。

- Lifecycle は「何日たったら移すか」を先に決める
- Intelligent-Tiering は「どう読まれているか」を見て調整する

この違いを押さえないまま「どちらも自動で安くするもの」と覚えると、Glacier への移行後に即時参照できる前提で設計してしまうなど、危ない判断につながります。

## S3 Lifecycle の役割

S3 Lifecycle は、オブジェクトの経過日数などを条件にして、ストレージクラス移行や削除を自動化する仕組みです。

例えば次のようなルールを定義できます。

- 30 日後に Standard-IA へ移行する
- 90 日後に Glacier Flexible Retrieval へ移行する
- 365 日後に Deep Archive へ移行する
- 730 日後に削除する

このため、Lifecycle は **時間とともに確実に冷えていくデータ** に向いています。

### 向いている例

- 長期バックアップ
- 一定期間後はほぼ見ない監査ログ
- 保管年限が決まっているアーカイブ

### 注意点

- アクセス増加に応じて自動で元のクラスへ戻るわけではない
- Glacier 系へ移した場合は復元操作が必要になることがある
- 取り出し時間と取り出し料金も考慮が必要

Lifecycle は便利ですが、あくまで **決めたルールを実行する仕組み** です。賢く需要変化を読んでくれるわけではありません。

## S3 Intelligent-Tiering の役割

S3 Intelligent-Tiering は、オブジェクトのアクセス状況に応じて、S3 が自動でアクセス階層を最適化する仕組みです。

このため、頻繁に読まれる時期と、ほとんど読まれない時期が混在するデータに向いています。

### 向いている例

- いつ再参照されるか読めないユーザーデータ
- 普段は静かだが障害調査時に急に読むログ
- 案件や季節要因でアクセス量が変動するファイル

### 注意点

- 自動最適化には管理コストがある
- 明らかに長期アーカイブ前提のデータでは、Lifecycle の方が素直なことも多い
- 何でも Intelligent-Tiering に入れれば最適という話ではない

つまり Intelligent-Tiering は、**読まれ方の不確実性を吸収したいときの選択肢** です。

## 比較表で整理する

| 観点 | S3 Lifecycle | S3 Intelligent-Tiering |
|---|---|---|
| 判断基準 | 経過日数やルール | 実際のアクセス頻度 |
| 主な目的 | 計画的に冷やす | 読まれ方に応じて自動調整する |
| 自動で戻るか | 基本的に戻らない | アクセス階層を自動調整できる |
| 向くデータ | 古くなる時期が読める | アクセス変動が読めない |
| 典型的な失敗 | Glacier に落として即時参照を期待する | 何でも入れて管理費だけ増やす |

この表で見ると、両者は代替ではなく、判断軸が異なることが分かります。

## 実務ではどう使い分けるか

実務で特に重要なのは、次の 3 点です。

### 1. 古くなるタイミングが明確か

バックアップや法令保管データのように、時間経過で価値が下がることが分かっているなら Lifecycle が扱いやすいです。

### 2. 突然の再参照があり得るか

たとえば、通常は見ないが監査や障害対応で急に必要になるデータは、Lifecycle だけで Glacier に落とすと運用事故になりやすいです。こうしたケースでは Intelligent-Tiering の方が安全です。

### 3. 取り出し待ちを業務が許容できるか

Glacier Flexible Retrieval や Deep Archive を使うなら、必要時にすぐ読めない可能性を前提にしなければいけません。ここを確認せずに「安いから」で選ぶのは危険です。

## 典型的なハマりどころ

例えば、アクセスパターンが読めないユーザーファイル群に対して、単純な Lifecycle ルールで Glacier へ落としたとします。この場合、利用者が急に古いファイルを開こうとしたとき、即時に返せず、復元待ちや追加料金が発生する可能性があります。

逆に、7 年保管が決まっている監査アーカイブを全部 Intelligent-Tiering に入れると、ルールで十分なものにまで自動最適化コストを払うことになります。

重要なのは、**コストだけではなく、再参照の不確実性と取り出し要求を一緒に見ること** です。

## 試験での判断基準

AWS 認定試験では、次のように切り分けるとかなり安定します。

- 経過日数ベースで移行したい → Lifecycle
- アクセスパターンが不明で自動最適化したい → Intelligent-Tiering
- Glacier に移した後もすぐ読める前提で書かれている → その前提を疑う

選択肢は名前が似ていても、問われているのは「時間で決めるか」「アクセス実態で決めるか」です。

## まとめ

S3 Lifecycle と Intelligent-Tiering は、どちらも S3 のコスト最適化に有効ですが、役割は同じではありません。

- Lifecycle は、時間経過を基準に計画的に冷やす仕組み
- Intelligent-Tiering は、アクセス状況を見ながら自動で最適化する仕組み

そのため、

- 古くなる時期が読めるなら Lifecycle
- 読まれ方が読めないなら Intelligent-Tiering

という判断が基本になります。

S3 のコスト設計は、安さだけで選ぶと後で必ず取り出し要件に刺されます。Lifecycle と Intelligent-Tiering の違いを先に切り分けておくと、設計の精度がかなり上がります。
